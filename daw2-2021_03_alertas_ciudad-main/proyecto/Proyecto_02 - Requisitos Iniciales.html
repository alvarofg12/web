Proyecto en Grupo
2. Requisitos Iniciales.

   Se desea diseñar una aplicación web para la Publicación y Gestión de Alertas, Avisos, Sucesos, Eventos que ocurren en tu Ciudad o cercanías..

   La aplicación será un portal web en el que principalmente se podrán ver, buscar y consultar las Alertas que se activen en el sistema por cualquiera de los usuarios registrados o dados de alta en el sistema.
   Las búsquedas se deberán poder realizar mediante uno o varios filtros donde el resultado serán las alertas que cumplan el/los filtros. Los posibles filtros serán por etiquetas de alertas, clasificación de categorías, por fechas, por lugares, por estado, etc. El resultado se mostrará como una lista resumida de alertas, ordenada por fecha de forma predeterminada y paginada si son muchas. Podrá haber alertas que aún no sean visibles porque no están activadas o se han puesto invisibles por cualquier causa.
   Se podrá elegir una alerta para visualizar su ficha en detalle, con todos los datos (públicos) que contenga: título, descripción, fecha/hora de aparición, duración (si se conoce), etiquetas, categoría, ubicación o lugar donde ocurre, imágen o imágenes de la alerta, etc.
   La aplicación además permitirá a los usuarios registrados crear nuevas alertas, introducir comentarios a las alertas o respuestas a comentarios, mantener su propia ficha de registro, entre otras funciones.

Requerimientos en general:
  • Portada "Previa": Una página inicial sencilla en la que se mostrará el logotipo del portal y la información de la utilidad del mismo, así como una zona para poder acceder a la aplicación como usuario y/o poder registrarse. Se mostrará información de las últimas alertas, por ejemplo las 20 alertas más recientes por orden de descendiente fecha de llegada.

  • Portada "Inicial": Una página inicial más compleja en la que se mostrará el logotipo del portal, una zona con una barra de navegación por las diferentes opciones de la aplicación, una zona de acceso a la misma (Login / Registro / Logout), un menú de selección de clasificación de actividades a la izquierda (visible según tamaño de pantalla o dispositivo móvil), unas serie de bloques o zonas en la parte central con las "fichas resumen" de las N actividades más próximas en el tiempo, más recientes, más visitadas, más buscadas, más valoradas, recomendadas (del mes, de la semana), según geolocalización del navegador cliente del usuario, o cualquier tipo de control que sirva para recomendar las actividades a los usuarios. Además se tendrá una zona con el posible buscador de actividades para realizar filtros o un enlace para acceder a la zona de búsqueda. En la parte derecha se mostrarán una serie de anuncios o similar de los posibles patrocinadores del portal o de las actividades que se están mostrando en ese momento en la página.
    • Si se activa cualquier filtro de clasificación o de búsqueda, se podrá pulsar un botón de "Filtrar" o "Buscar" para que cambie el contenido central acorde con el filtro. En este caso, la lista de actividades resultantes deberá salir paginada con unas 10 "fichas resumen" por página, permitiendo movernos entre las posibles páginas resultantes. Se debe poder cambiar el número de resultados por página mediante un desplegable predefinido en el que se tendrán las opciones 10, 25, 50, 100 y Todas.
    • Si se desactiva un filtro que está activado, se volverá a mostrar la vista inicial o página principal de actividades recomendadas.

Requisitos de Información / Datos:
     *** Información y Datos que debe almacenar y manejar el sistema para proporcionar las funcionalidades indicadas en los requisitos.
  • Alerta: Elemento con → título, descripción, fecha/hora inicio, duración estimada, Lugar, detalles, notas, url externa (dirección web opcional), Imagen principal, [Imágenes adicionales,] Etiquetas, Categoría, Estados de alerta, Registro de usuario, notas administrador, Comentarios.

  • Lugar: Son las posibles ubicaciones. Sustituto de → dirección, notas del lugar, Área.

  • Área: Alguna de las formas de tener ubicaciones relacionadas:
    • Sustituto de → código de clase de área, [País | Estado | Provincia | Poblacion | Zona | ...].
      *** Se deben fijar desde el diseño clases de datos relacionadas entre sí de forma jerarquizada. En esta opción, se deben "identificar" las diferentes clases por un código de clase (número o letra) para la implementación en la base de datos, como por ejemplo: 1=Pais, 2=Estado, 3=Provincia, 4=Poblacion, 5=Zona, ...
      • País: nombre, código internacional de país, código idioma (si se considera útil para la identificación automática del idioma y localización)
      • Estado: nombre, País. *** Considerar el Estado como Comunidad Autónoma, Región o similar.
      • Provincia: nombre, Estado
      • Población: nombre, Provincia
      • Zona: nombre, Población
      • ...
    • Elemento con → código de clase de área (por ejemplo: 0=Planeta, 1=Continente, 2=Pais, 3=Estado, 4=Region, 5=Provincia, 6=Municipio, 7=Localidad, 8=Barrio, 9=Zona), nombre, Área padre.
      *** Agrupar las posibles áreas en un conjunto en el que la jerarquía se realiza por auto-relaciones, y donde un elemento de clase N sólo puede estar relacionado con un elemento padre de clase <N, o lo que es lo mismo, o N-1, o N-2, ..., o 1, o 0 (CERO).
      *** En este caso, los códigos de clase se pueden fijar desde un primer momento y tener la lista disponible desde código o como una enumeración en la base de datos.

  • Imagen: Sustituto de → nombre (identificador de archivo externo en carpeta predefinida -ruta escribible por el servidor web-) o campo binario con la imagen subida, revisada (lógico), Registro de Usuario, notas administrador.

  • Imágenes: Elemento con → Imagen, Actividad. *** Relaciona una imagen con una actividad con lo que se pueden tener múltiples imágenes por actividad.

  • Etiquetas de Alerta: Sustituto de relación que vincula una etiqueta con una alerta, puede ser una de la opciones:
    • Elemento con → Etiqueta, Alerta. *** Relaciona ambas clases de objeto. En este caso debe definirse:
      • Etiqueta: Elemento con → nombre. *** Un conjunto de palabras clave utilizadas para identificar y clasificar de diferentes maneras las alertas.
    • Sustituto de → etiquetas_id. *** Una lista de identificadores de Etiqueta con los que se relaciona la alerta.
    • Sustituto de → etiquetas_nombre. *** Una lista de datos "nombre" de las propias etiquetas con las que se relaciona la alerta.
    *** En el caso de usar listas serían cadenas de datos separados por comas, teniendo especial cuidado en que la coma no aparezca en el propio contenido de los datos.
    *** Para gestionar adecuadamente las etiquetas de nueva creación propuestas por los usuarios, se generará una incidencia para los moderadores o administradores que indique la etiqueta creada, la fecha/hora del momento y el usuario que lo ha realizado para que tengan constancia de ello y se pueda revisar / actuar en consecuencia.

  • Categoría: Por una parte, desde el punto de vista de la alerta, es un sustituto de relación que vincula a la alerta con una clasificación de ellas, dadas por el conjunto de Categorías configuradas por los administradores del sistema. Por otra parte, desde el punto de vista de las propias Categorías, es un elemento de un conjunto jerarquizado de agrupación/categoría de etiquetas elaborado por los administradores del portal, el cual sería un elemento con → nombre, descripción, Categoría padre. Para identificar la propia categoría y posteriormente incluir en ella las etiquetas relacionadas:
    • Etiquetas de Categoría: Elemento de relación entre una Categoría y una serie de etiquetas, con → Categoría, Etiqueta.

  • Categorías: Conjunto de categorías dentro del sistema. Es una serie de elementos de tipo → Categoría.

  • Estados de Alerta: Sustituto de → activada (lógico), visible (lógico), terminada (0:No, 1:Realizada, 2:Suspendida, 3:Cancelada por inadecuada), fecha/hora de terminación, motivo de terminación, número denuncias, fecha/hora de la primera denuncia, bloqueada (0:No, 1:Si,bloqueada por denuncias, 2:Si,bloqueada por administrador, 3:Si,bloqueada por moderador), Usuario bloqueador (o NULO si no hay o lo hizo el sistema automáticamente), fecha/hora del bloqueo, motivo del bloqueo.

  • Comentarios: Sustituto de relación que vincula una alerta con una serie de comentarios realizados por los usuarios del sistema. Es un elemento con → Alerta, Comentario.

  • Comentario: Sustituto de → Registro de usuario, texto, Comentario padre, cerrado (lógico), número denuncias, fecha/hora de la primera denuncia, bloqueado (0:No, 1:Si, por denuncias, 2:Si, por administrador, 3:Si, por moderador), Usuario bloqueador, fecha/hora del bloqueo, motivo del bloqueo.

  • Registro de usuario: Sustituto de → fecha/hora de creación, Usuario creador, fecha/hora de modificación, Usuario modificador.
    *** Sirve tanto para relacionar algún elemento de información con los usuarios que lo han creado y/o modificado, como para conocer el momento de cuando lo han realizado.

  • Usuario: Elemento con → email, password, nick, nombre, apellidos, fecha_nacimiento (opcional), dirección (opcional), Área de localización (se recomienda obligatoria), rol (N:Normal, M:Moderador, A:Administrador), fecha/hora de registro, registro confirmado, ultimo acceso, intentos acceso, bloqueado (0:No, 1:Si, por fallos de acceso, 2:Si, por administrador, 3:Si, por moderador), Usuario bloqueador, fecha/hora del bloqueo, motivo del bloqueo, Áreas de moderación, Comentarios, Alertas, Incidencias.
    • Áreas de moderación: Sustituto de relación que vincula un usuario con un área de influencia para actuar como moderador dentro de ella. Se utiliza a la hora de aplicar/validar los permisos de acceso a las opciones de moderación por áreas. Es un elemento con → Usuario, Área.
    • Comentarios, Alertas: Datos/Tablas relacionadas ya comentadas más arriba.

  • Incidencias: Sustituto de relación que vincula un usuario con una notificación dentro del sistema, y opcionalmente con un usuario destino, Alerta, Comentario. Es una serie de elementos de tipo → Incidencia.

  • Incidencia: Elemento con → fecha/hora, código de clase de aviso (A:Aviso, N:Notificación, D:Denuncia, C:Consulta, M:Mensaje Genérico, ...), texto, Usuario destino (NULO si es para un moderador o administrador y no se conoce su usuario o aún no ha sido aceptada la incidencia por uno de ellos), Usuario origen (NULO si es generada por el sistema), Alerta (NULO si no está relacionada con una alerta), Comentario (NULO si no está relacionada con un comentario), fecha/hora de lectura (NULO si no se ha leído o se ha desmarcado como leído), fecha/hora de borrado (NULO si no se ha borrado), fecha/hora de aceptación (NULO si no se ha aceptado para su gestión por un moderador o administrador, no usado en otros usuarios).
    *** Una incidencia pasa a estar "leída" si el usuario destino la abre estando a NULO la fecha/hora de lectura. Si el usuario lo desea, puede cambiarlo a "no leída"; en este caso se establace a NULO el dato.
    *** Una incidencia pasa a estar "borrada" si el usuario destino la marca como tal, o se realiza el proceso de pre-borrado automático según los días que han transcurrido desde su lectura. No podrá pasar a borrada si la incidencia no está aceptada en caso de ser para un moderador o administrador. En este estado la incidencia se eliminará definitivamente del sistema cuando se realice el proceso de borrado automático y hayan transcurrido los días configurados para ello. Mientras una incidencia no se borre definitivamente, el usuario podrá cambiar su estado a "no borrada"; en este caso se establace a NULO el dato de la fecha/hora de borrado.
    *** Una incidencia pasa a estar "aceptada" si el usuario destino es un moderador o administrador que la marca para gestionarla. En el caso de que el usuario destino sea NULO, cualquier moderador o administrador de la zona de influencia de la incidencia podrá aceptarla, momento en el que se completará el dato con el usuario que la ha aceptado. Si una incidencia no ha sido aceptada, no podrá ser marcada como borrada.

  • Configuraciones: Será un elemento con → variable, valor. Concretamente un conjunto de información que servirá para establecer datos más o menos fijos (invariables en el tiempo) que se utilizan en la aplicación durante su ejecución. Por ejemplo...
    • numero_alertas_portada: el número máximo de alertas más relevantes para los bloques de la página principal,
    • numero_lineas_pagina: el número máximo de líneas por página de datos en las diferentes gestiones,
    • numero_denuncias_alerta: el número mínimo de denuncias para una alerta antes de ser bloqueada,
    • numero_denuncias_comentario: el número mínimo de denuncias para un comentario antes de ser bloqueado,
    • numero_intentos_usuario: el número mínimo de intentos de acceso de un usuario antes de ser bloqueado,
    • dias_incidencias_leidas: el intervalo de tiempo en días desde que una incidencia se marca como leida hasta que se marca como borrada,
    • dias_incidencias_borradas: el intervalo de tiempo en días desde que una incidencia se marca como borrada hasta que se elimina definitivamente de la base de datos.
    • ...

  • Logs: Elemento con → fecha/hora, clase del nivel de log (E:Error, A:Aviso, I:Información, D:Depuración), modulo aplicación, texto.
